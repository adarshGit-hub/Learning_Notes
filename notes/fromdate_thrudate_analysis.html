Part 3 · T-1 Impact on Datamodel & Business Processes

⚠️

The fromDate/thruDate pattern implements temporal versioning. Instead of updating a record in-place (losing history), a new record is created with a new fromDate, and the old record's thruDate is set to "now". This preserves full audit history of every change.

Pattern Rules

  • fromDate as PK → The entity supports multiple concurrent valid records for the same logical relationship, differentiated by time
  • thruDate = NULL → Record is currently active/valid
  • thruDate = <timestamp> → Record was expired/superseded at that time
  • New record → A new row with fromDate = now and thruDate = NULL replaces the old active record

How This Changes Business Processes

1. Order Identification Lookup

When system needs to find an order by its Shopify ID:

Without Date FilterWith Date Filter
Returns expired/old mappings if a Shopify order ID was remappedReturns only the current valid mapping

2. Product SKU Lookup (GoodIdentification)

A product's SKU can change over time — new GoodIdentification row inserted, old one expired. Must filter by date to get the current SKU→productId mapping.

3. Customer Address Changes (PartyContactMech)

When a customer updates their address: old PartyContactMech gets thruDate = now, new one inserted with fromDate = now. Ship-to resolution must use date filter to get the current address.

4. Facility-Store Mapping (ProductStoreFacility)

When a facility is added/removed from a store, old mapping gets expired. During brokering, only currently-active facilities should be considered.

5. Product Category Membership

Pre-order/backorder detection relies on category membership. A product can be moved in/out of pre-order category — must check current membership, not historical.

Code Patterns for Correct Date-Filtered Fetching

The codebase uses two primary patterns to ensure only current (non-expired) records are fetched:

Pattern 1: filterByDate()
Post-query in-memory filter
.list()
.filterByDate(
  "fromDate",
  "thruDate",
  ec.user.nowTimestamp
)
Pattern 2: conditionDate()
Query-level DB filter (preferred for large datasets)
.conditionDate(
  "fromDate",
  "thruDate",
  null  // null = now
)
.list()

Pattern 1 — Real Examples

What Goes Wrong Without Date Filtering?

🚨

If filterByDate / conditionDate is NOT used on temporal entities, the following bugs occur:

ScenarioBugImpact
Customer changed address, order importedOld address fetched for ship-toOrder shipped to wrong address
Product SKU changed, return createdOld product resolved from SKUWrong product on return item
Facility removed from store, brokering runsRemoved facility still consideredOrders allocated to inactive facility
Product moved out of pre-order categoryStill treated as pre-orderOrder stuck in Pre-Order Parking
Product variant association changedOld component products usedWrong items in marketing package
Return re-identified with new Shopify IDDuplicate return createdFinancial reconciliation errors
💡

Rule of thumb: Any time you query an entity with fromDate/thruDate and want the current record, use:
• .list().filterByDate("fromDate","thruDate", timestamp) — in-memory post-filtering
• .conditionDate("fromDate","thruDate", timestamp) — DB-level (preferred for large datasets)
Pass null as timestamp to default to "now".

Key Takeaways

Takeaway 01

fromDate/thruDate is NOT just metadata — it is a core part of the Primary Key in most temporal entities, meaning the same logical relationship can have multiple rows differentiated only by time.

Takeaway 02

Every user action that "changes" a temporal relationship creates a new row (fromDate=now) and expires the old row (thruDate=now), rather than updating in place.

Takeaway 03

Data fetching code MUST always use filterByDate or conditionDate when querying temporal entities to avoid stale/expired data bugs.

Takeaway 04

Two patterns exist: filterByDate (post-query, in-memory) and conditionDate (query-level, DB-side). The latter is more efficient for large datasets.

Takeaway 05

The most impactful temporal entities in order/return flows: GoodIdentification, ReturnIdentification, OrderIdentification, ProductAssoc, PartyContactMech, ProductStoreFacility, ProdCatalogCategory.
